# Phase 5.1 Plan 07: Per-Product Essentials Presets + Extended Compile/Loader Summary

**One-liner:** Ships 5 flagship per-product essentials preset JSONs
(`powerbi`, `pwrapps`, `pwrauto`, `exo`, `sp-admin`) plus extensions to
`compile-preset.mjs` and `preset-loader.ts` so all 6 presets (legacy
`essentials-v1` + 5 new) compile in one pass, resolve through a single
`presetFor(version)` entry point, and validate against the fresh tool
registry with product-prefix + typo-resistance contracts.

## What Was Built

### Task 1 — 5 per-product preset JSON files

- **`src/presets/powerbi-essentials.json`** (12 flagship `__powerbi__*` ops):
  - Tenant admin: `Admin_GetActivityEvents`, `Admin_GetCapacitiesAsAdmin`
  - Admin inventories: `Apps_GetAppsAsAdmin`, `Groups_GetGroupsAsAdmin`,
    `Datasets_GetDatasetsAsAdmin`, `Reports_GetReportsAsAdmin`,
    `Dashboards_GetDashboardsAsAdmin`, `Dataflows_GetDataflowsAsAdmin`,
    `Pipelines_GetPipelinesAsAdmin`
  - Delegated ops: `Capacities_GetCapacities`, `Gateways_GetGateways`, `Groups_GetGroups`

- **`src/presets/pwrapps-essentials.json`** (10 flagship `__pwrapps__*` ops):
  Apps (3), environments (2), connections (2), connectors (1), licenses + users (2).

- **`src/presets/pwrauto-essentials.json`** (12 flagship `__pwrauto__*` ops):
  Flows (2), flow runs (3 incl. cancel), environments (2), connections (2),
  owners + approvals (3).

- **`src/presets/exo-essentials.json`** (8 flagship `__exo__*` ops):
  Tenant config (2), mailboxes (Get + Set), folder permissions (Get + Add),
  distribution groups (static + dynamic). Includes `set-mailbox` because it
  is Microsoft's own REST v2 flagship write.

- **`src/presets/sp-admin-essentials.json`** (12 flagship `__spadmin__*` ops):

  storage (2). `sp-admin` is the Product enum member; prefix literal is
  `__spadmin__` (no dash, per VALID_PREFIX_RE).

Every preset JSON carries:
- `version` literal matching filename stem
- `product` enum member + `prefix` literal
- `sections` breakdown (documentation only)
- `rationale` block with one-line justification per op
- `ops` array of prefixed aliases (8-15 per preset)

Longest alias: `__exo__get-dynamic-distribution-group-member` (44 chars,
well under the MCP 64-char cap).

### Task 2 — `bin/modules/compile-preset.mjs` extended

Refactored `compileEssentialsPreset(generatedDir, presetsDir)` to compile
the legacy `essentials-v1` preset AND the 5 new per-product presets in a
single pass:

- **`PRESET_SPECS` frozen table** drives per-preset constraints:
  `{version, filename, constName, exactCount, prefix}`. One source of truth
  for every validator knob.
- **`essentials-v1` remains MANDATORY** — absent → hard error (same as before).
  The legacy 150-op bound is preserved via `exactCount: 150`.
- **Per-product presets are OPTIONAL** — absent files are silently skipped.
  Environments that predate plan 05.1-07 can compile the legacy preset alone.
- **Per-preset validation** added for:
  - non-empty `ops` array
  - no duplicate ops
  - every op carries the product's `__<prefix>__` literal (prefix `null` skips
    this check for `essentials-v1` which is intentionally cross-product)
  - every op present in the fresh registry (T-05-06 / T-5.1-04 typo resistance)
- **Emitted `generated-index.ts`** carries up to six `ReadonlySet<string>`
  `Object.freeze`d exports + a 6-entry `PRESET_VERSIONS` map.
- **Public signature + return shape preserved** — existing orchestrator
  (`bin/generate-graph-client.mjs` Step 5) and test stubs need zero edits.
- **`getPresetSpecs()` exported** for tests + future introspection.

- **`src/presets/generated-index.ts`** extended with 5 empty-stub

## Why It's Shaped This Way

**Why one preset JSON per product (vs. a single multi-preset file):** Each
JSON maps 1:1 to a `PRESET_SPECS` entry, keeps the diff locality clean when
a preset evolves, and lets per-product plans grow independently without
file-level merge conflicts. The existing `essentials-v1.json` pattern is
the precedent.

**Why `PRESET_SPECS` as a single frozen table (vs. separate
per-preset functions):** Every per-preset constraint (version literal, const
name, length bound, product prefix) is tabular by nature. Centralizing
keeps the validator loop short (iterate + validate each entry) and gives
tests a single introspection point via `getPresetSpecs()`. Adding
`essentials-v2` or per-product-v2 is a one-line append.

**Why per-product presets are optional from the filesystem's
perspective:** Environments that predate plan 05.1-07 can compile the
legacy preset alone. This lets the orchestrator's `Step 5` call site stay
identical while per-product JSONs land incrementally. `essentials-v1`
remains mandatory because it is the fresh-tenant default — its absence is
always a mistake.

**Why the stub `generated-index.ts` carries empty frozen Sets for the 5
new presets:** preset-loader.ts imports them at TypeScript compile time.
Before the next codegen run they need to resolve to SOMETHING, and empty
`Object.freeze(new Set())` placeholders are the minimum viable resolution
that preserves fail-closed (unknown preset → zero tools). The compile step
atomically overwrites this entire file on the next `npm run generate`, so
stubs never linger in production.

**Why `DEFAULT_PRESET_VERSION` stays `essentials-v1`:** Plan 05.1-07 adds
5 opt-in per-product presets without changing the cross-product Graph
default. A fresh tenant still receives 150 Graph v1.0 ops by default;
operators must explicitly PATCH `preset_version` or `enabled_tools` to
switch to a per-product preset. This keeps the tenant onboarding contract
stable and prevents accidental tool-surface expansion for existing tenants.

**Why 12 ops for Power BI, 10 for Power Apps, 12 for Power Automate, 8 for
Exchange, 12 for SharePoint:** Each preset covers its product's read-first
admin workflow breadth with 1-2 flagship writes where admins genuinely ask
for them. Under the 8-15 op window means each preset stays well under the
150-op legacy bellwether. Exchange at 8 reflects the smaller 10-op total
surface (set-mailbox is included because Microsoft's own REST v2 docs lead
with it). Longest alias (`__exo__get-dynamic-distribution-group-member`
= 44 chars) is well under the MCP 64-char cap.

**Why the product-prefix invariant is enforced at compile (vs. trusted at
edit time):** Catches the typical mistake of pasting a cross-product alias
into the wrong preset — e.g., a `__pwrapps__*` alias landing in the
powerbi-essentials.json. The test `rejects a per-product preset whose op
is missing its product prefix` pins this. Error preview truncated at 5
offending aliases for readable compile failures.

**1. [Rule 1 - Test emit format assumption] Compile output uses JSON.stringify quoting**

- **Found during:** Task 4 test run
- **Issue:** Initial test assertions checked `expect(out).toContain("'essentials-v1'")`

  Test now passes deterministically regardless of downstream formatter behavior.
- **Files modified:** `test/bin/per-product-preset-compile.test.mjs`
- **Commit:** `d369c6b` (bundled with the Task 4 test landing)

## Known Stubs

**Intentional stubs in `src/presets/generated-index.ts`:**

The 5 new per-product preset constants (`POWERBI_ESSENTIALS_OPS` /
`PWRAPPS_ESSENTIALS_OPS` / `PWRAUTO_ESSENTIALS_OPS` / `EXO_ESSENTIALS_OPS` /
`SP_ADMIN_ESSENTIALS_OPS`) currently land as empty `Object.freeze(new Set<string>([]))`
placeholders. This is NOT a functional stub — it is a compile-time
resolution shim. The first full codegen run under `MS365_MCP_FULL_COVERAGE=1`
atomically overwrites this file with real alias sets sourced from the 5
JSONs and validated against the fresh registry.

**Why this is safe:**
- Empty Sets preserve fail-closed (unknown preset → zero tools, never wider).
- `presetFor('powerbi-essentials')` on a stub returns an empty frozen Set;
  `Set.has(alias)` yields false for every call — same as if the preset didn't
  exist. No accidental tool expansion.
- Tests verify stub-safety at `test/tool-selection/preset-loader-extended.test.ts`
  (`all 5 per-product re-exports are ReadonlySet instances (stub-safe)`).
- Production deploys run codegen at build time (`Dockerfile` runs
  `npm run generate` before `npm run build`), so stubs are always overwritten
  before any tenant reaches dispatch.

**Hand-offs (NOT stubs):**
- Admin PATCH validators (future plan) should source preset name literals
  from `KNOWN_PRESET_VERSIONS` instead of hard-coding strings — the
  single source of truth is now in preset-loader.ts.
