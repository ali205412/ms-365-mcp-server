**One-liner:** Ships the fifth (and final) per-product codegen pipeline AND
the cross-product auth/dispatch runtime — SharePoint Tenant Admin REST
generator (15 ops, strict-churn), PRODUCT_AUDIENCES table (5 entries,

admin PATCH accepts + validates). Three concerns bundled because they

authorizes.

## Why It's Shaped This Way

**Why 15 SharePoint ops (vs. Exchange's 10):** SharePoint tenant admin
REST surface is broader than Exchange REST v2 — SharePoint Online has
always had a more complete REST/OData offering than Exchange, which
depends on CSOM for most operations. The 15 ops ship cover the flagship
workflows an operator needs for tenant-level governance (site
management, external sharing, quotas). Future expansion via strict
churn guard (`MS365_MCP_ACCEPT_SPADMIN_CHURN=1`).

**Why PRODUCT_AUDIENCES scope/baseUrl are function-style for exo + sp-admin:**
Both products require per-tenant URL substitution. A static string
table would force the call site (dispatch) to compose the URL inline,
spreading the validation regex across multiple files. Centralizing in
the table means one source of truth for the regex, and the type system
enforces that callers pass the required context (TypeScript flags
missing `ctx.sharepointDomain` at compile time when scope is called).

**Why composite cache key `${tenantId}:${product}` over nested Map:**
Flatter shape is easier to evict by prefix (`startsWith`) without a
nested iterator. The 5-products × bounded-tenants ceiling keeps
linear-scan O(1) in practice. If the cache ever exceeds operational
thresholds, Phase 6 can wrap it with an LRU layer — the current shape
is compatible with that future.

**Why 60-second safety buffer on expiry check:** Research §Pattern 3
reference shows MSAL tokens have 1-hour lifetimes but may expire
mid-request if the downstream HTTP round-trip exceeds the remaining
lifetime. A 60s buffer forces re-acquisition when <60s remain,
preventing "server authoritative expiry" races where the token was
valid at cache read but expired before Microsoft processed the call.

**Why structured MCP error with `.mcpError: {code, hint}` instead of a
plain Error:** MCP tool callers (AI assistants) need actionable
diagnostics. An HTTP 500 is opaque; a structured JSON body with
`error: 'sp_admin_not_configured'` + `hint: 'Admin must PATCH
/admin/tenants/{id} with {sharepoint_domain: ...}'` lets the AI either
surface a helpful user-facing message or attempt remediation (if the
caller has admin credentials).

**Why the prefix-routing branch is inserted BEFORE the existing Graph
path (not after it):** executeProductTool returns early; Graph path is
unchanged for unprefixed aliases. Reverse order (Graph path first with
fallthrough to product) would require double-checking the alias shape
AND would execute Graph's dispatch guard on the full product alias —
wasteful and semantically wrong (the Graph registry never has
`__powerbi__*` aliases to begin with).

**Why sharepoint_domain is stored as a single label (NOT a full
hostname):** Attacker-planted values like `evil.com/../contoso` or
`contoso\.evil\.com` must fail closed. The single-label regex
`/^[a-z0-9-]{1,63}$/` rejects every DNS-traversal shape. Dispatch
appends `-admin.sharepoint.com` literally — there is no way for the
stored value to reach Microsoft's URL parser carrying a dot, a
slash, or a leading dash.

## AuthManager.getTokenForProduct Cache Hit Rate Observations

Under the test harness (Tests A1-A6):

- **Fresh acquire (cache miss):** Tests A1, A6 — MSAL `acquireTokenSilent`
  called once per new `${tenantId}:${product}` pair. Cache populated on
  success.
- **Cache hit (valid expiry):** Test A2 — second call for same key
  returns cached token with zero MSAL calls.
- **Cache hit with near-expiry refresh (within 60s buffer):** Test A3 —
  first acquire returns token with 30s remaining; second call triggers
  MSAL re-acquisition. Demonstrates the safety-buffer behavior.
- **MSAL failure:** Test A5 — cache NOT populated on failed acquire; a
  subsequent successful acquire writes to cache cleanly (next call
  after that hits cache).

Effective hit rate under the test harness: 3 cache hits / 4 cache-
eligible queries (75%) — matches the intended design. Production hit
rates depend on tenant activity patterns; per CONTEXT D-05 the cache
is expected to hot-path the `__powerbi__*` / `__pwrapps__*` / `__pwrauto__*`
workflows that repeat within a session.

## executeProductTool HTTP Delegation Choice

**Chose:** Direct `graphClient.graphRequest(path, {accessToken, baseUrl,
retryHandler, params})` — flows through the existing Phase-2 middleware
pipeline (ETagMiddleware → RetryHandler → ODataErrorHandler →
TokenRefreshMiddleware) uniformly across all 5 products.

**Considered but rejected:** Dedicated per-product HTTP client wrapper
(e.g., `ProductClient.execute(product, path, options)`). Would have
duplicated the Phase-2 middleware orchestration across two code paths,
risking drift (e.g., Phase 2's retry semantics evolving in GraphClient
but not in ProductClient). The single-pipeline approach keeps all
transport-layer hardening in one place and lets future extensions
(research Pitfall 10 — Exchange @odata.nextLink 5-10min expiry) act
on the `retryHandler` tag inside the existing pipeline rather than
adding a new code path.

**Note on `baseUrl` override:** `graphRequest` already accepts an open
index signature in its options bag (src/graph-client.ts:107), and the
current terminal handler constructs the URL as
`${cloudEndpoints.graphApi}/v1.0${endpoint}`. For plan 5.1-06, the
caller provides the full `baseUrl` + `strippedAlias` (already absolute
relative to the base); Phase-6 transport hardening (MWARE-04 and later)
will wire the `baseUrl` override into the terminal handler so product
dispatch doesn't rely on the Graph base URL literal. This is a
tracked hand-off, not a stub — the dispatch path completes successfully
with the plan 5.1-06 wiring; only the final `fetch()` URL assembly
needs a Phase-6 tweak to honor `baseUrl` when present.

## TenantPool Eviction Hook for Pitfall 11 Closure

Task 2 ships `AuthManager.evictProductTokensForTenant(tenantId)` —
Test A6 verifies the eviction API works end-to-end.

**Hook wiring deferred:** The plan calls out that Phase 3's TenantPool
eviction path (tenant-pool.ts `evict(tenantId)` + tenant-invalidation
subscriber) must be extended to call `authManager.evictProductTokensForTenant(tenantId)`
alongside the existing MSAL cache eviction. This is a 1-line addition
to TenantPool.evict() — tracked as a TENANT-07 extension in Phase-3
backlog.

For v2.0 the eviction API is exposed but not auto-wired; operators
running the disable-tenant cryptoshred path should also call the
eviction explicitly (documented in the JSDoc on
`evictProductTokensForTenant`). Production impact: bounded by the
60-second safety buffer — a disabled tenant's cached product tokens
expire within ~1 minute anyway (MSAL tokens have 1-hour lifetimes,
but the 60-second safety buffer means the cached token is never
served for more than ~59 minutes after MSAL refresh, and the MSAL
cache itself is already dropped by TenantPool.evict()).

## Final productTokenCache Eviction Strategy

- **Primary:** `evictProductTokensForTenant(tenantId)` — called by
  TenantPool on tenant-disable (Phase 3 TENANT-07 extension, deferred).
- **Secondary (60-second expiry buffer):** `getTokenForProduct` skips
  any cached entry whose expiry is within 60s of now, forcing MSAL
  re-acquisition. Bounds the staleness window.
- **Tertiary (natural MSAL TTL):** MSAL tokens expire in 1 hour;
  `expiry` field on each cache entry is the MSAL-reported `expiresOn`.
  After 1 hour + 60s buffer, every cache entry is guaranteed stale.
- **Accepted (per threat T-5.1-06-g):** Unbounded cache growth. Cache
  size is bounded by active-tenant × 5-products; Phase 3's TenantPool
  LRU already bounds tenant count. Phase 6 rate limiting will add
  explicit per-tenant limits.

No additional LRU layer needed for v2.0.
