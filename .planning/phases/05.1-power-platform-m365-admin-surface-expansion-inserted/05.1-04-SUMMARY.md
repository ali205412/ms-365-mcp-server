# Phase 5.1 Plan 04: Power Automate (Flow) Generator (`__pwrauto__` aliases) + Hand-Authored OpenAPI Fragment Summary

**One-liner:** Ships the third per-product codegen pipeline for Phase 5.1 —
a hand-authored OpenAPI 3.0 fragment covering 20 flagship Power Automate
Flow management ops (with Flow DSL `definition` + `connectionReferences`
typed as open objects to accommodate Microsoft's recursive Logic Apps
expression language), a `runPowerAutomatePipeline` wrapper over plan
05.1-01's shared `runProductPipeline` contract, and `__pwrauto__*`-
prefixed aliases under permissive churn.

## Why It's Shaped This Way

**Why `specUrl: null` at the wrapper (not `undefined` or omitted):**
Plan 05.1-01's `runProductPipeline` treats both `null` and `undefined`
as the hand-authored-spec branch. Matches Power Apps precedent — the
wrapper commits to the literal `null` to make the contract explicit at
the API boundary. Test 14 pins this so regressions show up as mock
assertion failures. Grepping for `specUrl: null` turns up every
hand-authored-spec consumer in one scan.

**Why 20 ops (not exactly 18 or the plan's required 18):** The plan
window was 15-25 with "target 18-20" recommendation. The plan's required
list is 18; adding `list-approvals` + `get-approval` rounds out the
approvals-workflow surface (callers frequently use approvals to gate
Flow execution against human review). Both are natural companions to
ops the plan already required (approvals hang off the same
`/environments/{envId}/...` root as Flow runs); preset decisions are
deferred to plan 5.1-07 regardless. Still within the 15-25 ceiling with
5-op headroom.

**Why the Flow DSL escape hatch (definition + connectionReferences as
`additionalProperties: true`) rather than a partial schema:** Microsoft's
Logic Apps DSL is a recursive expression language. A complete Zod schema
would need `z.lazy(() => ...)` combined with a closed union over all
trigger types + all action types + all expression types. Even a minimal
subset (`triggers`, `actions`, `inputs`, `outputs`) is recursive — an
action's `inputs` can contain an expression that references another
action's `outputs`. Partial coverage risks silently rejecting valid Flow
definitions at the gateway, which would break every Power Automate
caller whose Flow uses a trigger/action/expression shape we didn't
model. The escape hatch forwards opaque JSON to Microsoft's Flow runtime
(which owns DSL validation), mirroring the pattern Phase 5 used for
Graph `extensions` dynamic properties. Documented accept per research
§Anti-Patterns.

**Why the `servers:` field points at
`service.flow.microsoft.com/providers/Microsoft.ProcessSimple` even
though runtime dispatch ignores it:** Same rationale as Power Apps
(plan 5.1-03). Keeping it in the spec serves two purposes: (1)
documentation — operators reading the YAML see the canonical management
plane URL without consulting the plan 5.1-06 audiences table; (2)
openapi-zod-client emits relative paths regardless of `servers:`, so
the codegen output is unchanged either way. Threat T-5.1-04 mitigation
holds because plan 5.1-06's dispatch reads
`PRODUCT_AUDIENCES.get('pwrauto').baseUrl`, NOT this field — a rogue
edit to `servers:` cannot redirect live traffic.

**Why NO fixup needed in orchestrator integration tests (diff with
5.1-03):** Plan 5.1-03 added `runProductPipelines: async () => {}`
stubs to `test/bin/beta-churn-guard.test.mjs` (Test 5) and
`test/bin/coverage-check-orchestrator.test.mjs` (Tests 1, 3, 4) because
the real `runProductPipelines` iterator walks `PRODUCT_PIPELINES` and
calls each `run()` which throws `/hand-authored spec not found/i` when
the tmpdir has no staged spec. Those stubs are generic — they bypass
the iterator entirely, so they cover every product whose pipeline uses
the `specUrl: null` code path (Power Apps, Power Automate). The stubs
added by plan 5.1-03 fully cover this plan's new registration; no
additional test edits needed. Verified by the full `test/bin/` suite
going from 208/208 pass (pre-plan) to 223/223 pass (post-plan).

### Flow DSL escape hatch — observed codegen emission

Against the mini fixture's `trigger-flow` op with
`additionalProperties: true` request body, openapi-zod-client emits:

```typescript
schema: z.object({}).partial().passthrough().passthrough(),
```

The double `.passthrough()` is because openapi-zod-client emits
`.strict()` and the post-processor rewrites `.strict()` → `.passthrough()`
(which adds a `.passthrough()` chain onto the existing one emitted by
the generator). The test regex `/z\.object\(\{\}\)\.passthrough\(\)/`
matches the substring within the longer expression. Both
`z.record(z.any())` and `z.object({}).passthrough()` are valid
openapi-zod-client outputs for open objects; the observed output is the
latter form (more common in practice given our post-processor rewrite
chain).

Against the real 20-op spec, the `Flow.properties.properties.definition`
and `Flow.properties.properties.connectionReferences` schemas emit as:

```typescript
const Flow = z
  .object({
    ...
    properties: z
      .object({
        displayName: z.string(),
        state: z.enum(['Started', 'Stopped', 'Suspended']),
        definition: z
          .object({})
          .partial()
          .passthrough()
          .passthrough()
          .describe('Flow DSL JSON. Not schematized — recursive expression language. ...'),
        connectionReferences: z
          .object({})
          .partial()
          .passthrough()
          .passthrough()
          .describe('Connection refs map. Opaque; tenant-specific. ...'),
      })
      .partial()
      .passthrough()
      .passthrough(),
  })
  .partial()
  .passthrough()
  .passthrough();
```

Every recursive level of the Flow DSL body compiles to
`.partial().passthrough()` — exactly the escape hatch the plan
specified.
