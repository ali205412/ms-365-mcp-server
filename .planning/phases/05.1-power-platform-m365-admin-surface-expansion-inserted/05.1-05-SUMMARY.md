## Why It's Shaped This Way

**Why 10 operations, not 9 as research suggested:** The plan's research
document approximated "9 cmdlets" from admin-api-endpoints-reference,
but a closer reading shows MailboxFolderPermission supports 4 distinct
cmdlets (Get-, Add-, Set-, Remove-) not 3. Reconciling against
admin-api-endpoints-reference directly:

- Get-OrganizationConfig: 1
- Get-AcceptedDomain: 1
- Get-Mailbox, Set-Mailbox: 2
- Get/Add/Set/Remove-MailboxFolderPermission: 4
- Get-DistributionGroupMember: 1
- Get-DynamicDistributionGroupMember: 1

Total: 1+1+2+4+1+1 = **10** cmdlets across 6 endpoints. The plan's Task 1
step 1 caught this inconsistency inline and the spec ships 10 ops.

**Why STRICT churn for Exchange (vs. permissive for Power Platform):**
Exchange REST v2 is still-shipping public preview. Microsoft has already
expanded the surface twice in the last 18 months without announcement (per
research §Exchange REST v2 Shape). A silent addition could introduce new
auth scopes, CmdletInput body shapes, X-AnchorMailbox validation rules, or
even new dispatch-layer policies (e.g., Microsoft Teams admin REST v2
added `X-AnchorMailbox` REQUIRED without prior notice). Strict churn
forces operators to consciously review every delta via
`MS365_MCP_ACCEPT_EXO_CHURN=1`. The two-bucket preview shape (5 removed +
5 added per run-product-pipeline.mjs) makes the delta reviewable at a
glance.

**Why CmdletInput.Parameters = `additionalProperties: true` (not narrowed):**
Every Exchange cmdlet has its own parameter schema. Set-Mailbox alone
supports 400+ parameters across combinations. Modelling all of them would
triple the spec size, bloat the generated client.ts by 50+ KB, and
provide zero runtime safety — Microsoft's Exchange backend validates
parameters and returns structured errors. The preset layer (plan 5.1-07)
can narrow cmdlet-specific parameters per-op if operators demand it.
CmdletName match is a different problem — plan 5.1-06 dispatch validates
it against the op's expected cmdlet at request time (T-5.1-05-e).

**Why the `servers:` field points at
`https://outlook.office365.com/adminapi/beta/{tenantId}` even though
runtime dispatch ignores it:** Same rationale as Power Apps / Power
Automate. Keeping it in the spec serves two purposes: (1) documentation
— operators reading the YAML see the canonical REST v2 URL shape
without consulting plan 5.1-06; (2) openapi-zod-client emits relative
paths regardless of `servers:`, so the codegen output is unchanged
either way. Threat T-5.1-05-c + T-5.1-05-f defense-in-depth holds
because plan 5.1-06's dispatch reads
`PRODUCT_AUDIENCES.get('exo').baseUrl`, NOT this field — a rogue edit
to `servers:` cannot redirect live traffic.

**Why the test file (`exo.test.mjs`) is shipped in Task 1's RED commit,
not Task 2:** Matches the Power Automate precedent (plan 05.1-04 did the
same). Shipping all 21 tests upfront in Task 1 RED lets the 17 Task 1
tests verify artifact correctness the moment they land; the 4 Task 2
tests (18-21) fail with `Cannot find module '../../bin/modules/exo-admin.mjs'`
until Task 2 GREEN, which is the standard TDD flow.

**Why NO additional fixup needed in orchestrator integration tests:** Plan
5.1-03 already added `runProductPipelines: async () => {}` stubs to
`test/bin/beta-churn-guard.test.mjs` (Test 5) and
`test/bin/coverage-check-orchestrator.test.mjs` (Tests 1, 3, 4). Those
stubs bypass the iterator entirely so they cover ANY product whose pipeline
uses the `specUrl: null` code path (Power Apps, Power Automate, Exchange
Admin). No additional test edits needed. Verified by the full `test/bin/`
suite going from 119/119 pass (pre-plan) to 140/140 pass (post-plan).

## Integration Observations

### Exchange op count

The hand-authored spec has **10 operations** (deliberately exact — not a
ceiling). The pipeline emitted **all 10** `__exo__*` aliases on the
smoke-test integration run — no ops dropped by the simplifier, no
truncations required, no collisions against the seeded main client.ts.

### CmdletInput envelope — observed codegen emission

Against the mini fixture's `set-mailbox` op with the full CmdletInput
envelope, the emitted client.ts fragment contains the shape:

```typescript
body: z
  .object({
    CmdletInput: z
      .object({
        CmdletName: z.string(),
        Parameters: z.object({}).partial().passthrough(),
      })
      .passthrough(),
  })
  .passthrough(),
```

(Post-processor's `.strict()` → `.passthrough()` rewrite chain applied.)
The `CmdletInput` nesting, `CmdletName` string, and `Parameters` open
object all survive the pipeline. Plan 5.1-06 dispatch can inspect the
emitted schema to (a) locate the CmdletName string slot and (b) validate
caller-supplied CmdletName matches the per-op expected value.

### X-AnchorMailbox required header — observed codegen emission

Every emitted operation carries the `X-AnchorMailbox` header as an
openapi-zod-client `Header` parameter. Integration smoke test confirms:

```
X-AnchorMailbox emitted: true
```

— the header survives the post-process chain and is reachable by plan
5.1-06 dispatch for injection based on auth context (AAD-UPN for
delegated flows, APP:SystemMailbox constant for app-only global reads).

### Orthogonal alias namespace

The `__exo__` prefix is alpha-unique against every existing namespace
(v1 unprefixed, `__beta__`, `__powerbi__`, `__pwrapps__`, `__pwrauto__`).
No collisions possible by construction — the runProductPipeline
collision guard runs after prefix injection as a defense-in-depth check
anyway.

### Spec shape

Hand-authored spec is **15.1 KB** (openapi/openapi-exo.yaml); well below
the 100 KB informal threshold for T-5.1-05-e analogue (no DoS risk
during simplifier recursion). Every op takes X-AnchorMailbox (required)
+ common OData pagination params ($select, $filter, $top, $skip); path
params are typed `string`; POST/PUT request bodies use the shared
CmdletInputBody. Response payloads are typed as
`ExchangeResponse` (open object with optional `value[]` + `@odata.nextLink`).
