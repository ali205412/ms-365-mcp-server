# Phase 5.1 Plan 02: Power BI Generator (`__powerbi__` aliases) + Permissive Churn Guard Summary

**One-liner:** Ships the first per-product codegen pipeline for Phase 5.1 —
a Power BI wrapper over plan 05.1-01's shared `runProductPipeline`
contract, pinning microsoft/PowerBI-CSharp's Swagger 2.0 spec, emitting
285 `__powerbi__*`-prefixed aliases into `src/generated/client.ts`, and
enforcing a permissive churn policy per 05.1-CONTEXT D-04.

## What Was Built

**Task 1 — Power BI spec pinning + mini fixture + failing tests:**

- `openapi/openapi-powerbi.yaml` (907 KB, 288 operationIds) pinned via
  `npx @redocly/cli bundle` over microsoft/PowerBI-CSharp's upstream
  Swagger 2.0 JSON. Committed via a `.gitignore` allow-list
  (`!openapi/openapi-powerbi.yaml`) so `MS365_MCP_USE_SNAPSHOT=1` regens
  work offline. File header comment documents the conversion choice and
  refresh procedure.
- `test/bin/fixtures/mini-powerbi.yaml` — 3-operation OpenAPI 3.0 fixture:
  `WorkspacesList` (14 chars post-prefix), `WorkspacesGetDatasetsInGroupAsAdmin`
  (48 chars post-prefix, under the cap), and a 60-char operationId that
  forces sha1-8 truncation to exactly 64 chars.
- `bin/.last-powerbi-snapshot.json` — empty-baseline snapshot committed so
  fresh checkouts don't produce "missing snapshot" confusion on first run.
- `test/bin/powerbi.test.mjs` — 12 tests covering prefix invariant, 64-char
  truncation (both short-id pass-through and long-id truncation), permissive
  churn policy matrix (4 tests: removal without env / with env, addition
  without env, fresh-checkout), side-effect registration into
  `PRODUCT_PIPELINES`, idempotent double-import, path resolution, and the
  exact deps bag passed to `runProductPipeline`.

**Task 2 — Power BI wrapper + orchestrator registration:**

- `bin/modules/power-bi.mjs` — exports `POWERBI_SPEC_URL` (pinned to
  microsoft/PowerBI-CSharp raw URL), `POWERBI_PREFIX='__powerbi__'`,
  `POWERBI_SNAPSHOT_NAME='.last-powerbi-snapshot.json'`,
  `POWERBI_CHURN_ENV='MS365_MCP_ACCEPT_POWERBI_CHURN'`, and
  `runPowerBIPipeline({openapiDir, generatedDir, rootDir})`. On module
  load, side-effect-pushes `{name: 'powerbi', run: runPowerBIPipeline}`
  into the PRODUCT_PIPELINES registry, guarded by a
  `.some(entry => entry.name === 'powerbi')` idempotency check.
- `bin/modules/product-registry.mjs` (NEW, unplanned — see Deviations) —
  leaf module exporting `PRODUCT_PIPELINES`. Both the orchestrator and
  each per-product module import from this leaf instead of the
  orchestrator, breaking what would otherwise be an ESM circular import.
- `bin/generate-graph-client.mjs` — imports `PRODUCT_PIPELINES` from
  `./modules/product-registry.mjs`, re-exports it (backwards compat for
  any code that imports from the orchestrator), and adds one
  `import './modules/power-bi.mjs';` line for the side-effect registration.
  Plans 5.1-03..06 each add exactly one more such line.

## Why It's Shaped This Way

**Why a leaf `product-registry.mjs` module instead of keeping
PRODUCT_PIPELINES in the orchestrator:** Wave 1 defined the registry as
`export const PRODUCT_PIPELINES = []` inside `bin/generate-graph-client.mjs`.
Once this plan added `import './modules/power-bi.mjs';` at the top of
the orchestrator to trigger side-effect registration, the product module's
`import { PRODUCT_PIPELINES } from '../generate-graph-client.mjs'` fired
BEFORE the orchestrator's body ran — so the binding was in the ESM
temporal dead zone and the `.some()` call threw. Extracting the registry
into a leaf module (both `generate-graph-client.mjs` and `power-bi.mjs`
import from it) resolves the cycle cleanly and gives plans 5.1-03..06 a
single obvious import path.

**Why commit the YAML (converted) rather than raw swagger.json:** The
Power BI upstream is Swagger 2.0 JSON, and openapi-zod-client 1.18.3
accepts both 2.0 and 3.0. The YAML form was chosen for uniformity with
every other openapi/ file in the repo (Graph v1 + beta are committed as
YAML). The redocly bundle produces Swagger-2.0-flavoured YAML (it
preserves the `swagger: '2.0'` root field rather than upgrading to
OpenAPI 3), which openapi-zod-client handles natively — no dialect
surprises downstream.

**Why widen the prefix regex (deviation from Wave 1):** The `[a-z]` anchor
in run-product-pipeline.mjs was a carry-over from beta.mjs's T-05-03
mitigation. But T-05-03 is about rejecting already-prefixed aliases
(names starting with `_`), not about rejecting capitals. Every Microsoft
product this phase touches emits PascalCase operationIds. The widening
to `[a-zA-Z]` preserves the underscore-exclusion (what T-05-03 actually
needs) while covering the real-world casing. Wave 1's existing tests
(lowercase fixture) continue to pass, confirming backward compatibility.

## Integration Observations

### Power BI op count

The real spec contains 288 operationIds; the pipeline emits 285
`__powerbi__*` aliases. The 3-op gap is legitimate:

- 2 × `Datasets_SetAllDatasetConnections` (regular + `_InGroup` variant) —
  both marked `deprecated: true` in the upstream spec; openapi-zod-client
  drops deprecated ops by default. No action needed — the essentials
  preset (plan 5.1-06) wouldn't want these anyway.
- 1 × `1065e6a3-a020-4c0c-ada7-92b5fe99eec5` — a UUID-shaped operationId
  that starts with a digit. The prefix regex `[a-zA-Z]`-anchors skips
  numeric-first identifiers because they'd produce invalid JS identifiers
  after prefixing. This is correct behavior; the plan's "zero aliases"
  guard would fire if the spec were ALL numeric, but a single stray
  doesn't constitute spec-wide drift.

All 285 emitted aliases are under the MCP 64-char cap without truncation.
Longest: `__powerbi__WidelySharedArtifacts_LinksSharedToWholeOrganization`
(63 chars). This matches the research pin: "Power BI's longest op:
DataflowsGetUpstreamDataflowsInGroupAsAdmin (41 chars)" — with the
`__powerbi__` prefix (11 chars) the worst case tops out at 52-63 chars,
well under 64. The truncation path is still tested via the 60-char
fixture entry.

### Spec-format choice

Committed as **Swagger 2.0 YAML** (not upgraded to OpenAPI 3). redocly
bundle's YAML output preserved the 2.0 dialect, which openapi-zod-client
1.18.3 accepts natively. An upgrade to OpenAPI 3 would have added a step
to the refresh procedure (run redocly's `openapi-cli-converter` or
redocly's converter plugin) without behavioral benefit; deferred unless
a future plan has concrete need.

### No v1/beta alias collisions

runProductPipeline's collision guard didn't fire during the real-spec
integration run. The `__powerbi__` prefix makes genuine collision
essentially impossible (Graph's v1/beta aliases are unprefixed or
`__beta__`-prefixed; the namespace is orthogonal). The guard remains
important for future cases where a product might emit an alias that
post-truncation collides with another product's alias.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Wave 1 prefix regex too narrow for PascalCase**

- **Found during:** Task 1 RED — Power BI's fixture ops are PascalCase
  (`WorkspacesList`, `WorkspacesGetDatasetsInGroupAsAdmin`), but Wave 1's
  `[a-z]`-anchored regex in `run-product-pipeline.mjs` would prefix ZERO
  aliases and the zero-aliases invariant would throw before any test
  could exercise the integration path.
- **Issue:** Every Microsoft product operationId in this phase
  (Power BI, Power Apps, Power Automate, EXO REST v2, SharePoint Tenant
  Admin) starts with uppercase. A `[a-z]` anchor blocks the entire plan.
- **Fix:** Widened regex to `[a-zA-Z]` while keeping the underscore
  exclusion intact (T-05-03 security concern — "no bait-and-switch via
  upstream casing tricks" — is satisfied by the exclusion, not by the
  lowercase-only anchor).
- **Files modified:** `bin/modules/run-product-pipeline.mjs`
- **Commit:** `36ba3aa`

**2. [Rule 3 - Blocking] ESM circular-import between orchestrator and product modules**

- **Found during:** Task 2 GREEN — Tests 9 and 10 failed with "Cannot
  read properties of undefined (reading 'some')" because
  `PRODUCT_PIPELINES` was in the ESM temporal dead zone when
  `power-bi.mjs` ran its side-effect `.push()`.
- **Issue:** Orchestrator imports `./modules/power-bi.mjs` at top of file
  → product module imports `PRODUCT_PIPELINES` from the orchestrator →
  orchestrator's `export const PRODUCT_PIPELINES = []` body hasn't
  executed yet → TDZ error.
- **Fix:** Extracted `PRODUCT_PIPELINES` into a new leaf module
  `bin/modules/product-registry.mjs`. Orchestrator + product modules both
  import from the leaf. Orchestrator re-exports for backwards compat.
- **Files created:** `bin/modules/product-registry.mjs`
- **Files modified:** `bin/generate-graph-client.mjs`, `bin/modules/power-bi.mjs`
- **Commit:** `5729ff6`

**3. [Rule 2 - Missing functionality] `openapi/openapi-powerbi.yaml` gitignored**

- **Found during:** Task 1 — initial `git status` did not show the spec
  as untracked because `.gitignore` has `openapi` (whole directory
  ignored). The plan's success criterion "offline runs work under
  MS365_MCP_USE_SNAPSHOT=1" requires the spec committed.
- **Issue:** Without a `.gitignore` exception, the pinned spec can't be
  committed; `USE_SNAPSHOT=1` would fail on fresh checkout.
- **Fix:** Changed the rule from `openapi` (directory-level) to
  `openapi/*` + `!openapi/openapi-powerbi.yaml` so per-product specs can
  be individually pinned while the Graph v1/beta bulk specs stay
  local-only. Plans 5.1-03..05 will add additional allow-list lines for
  their committed specs.
- **Files modified:** `.gitignore`
- **Commit:** `7405862` (bundled with the RED test commit since the
  gitignore change was needed to make the spec stageable in the first place)

**4. [Rule 1 - Bug] Wave 1 test contract stale after registry population**

- **Found during:** Regression run after GREEN — Wave 1's
  `test/bin/generate-graph-client.test.mjs` Test D asserted
  `PRODUCT_PIPELINES.toHaveLength(0)`, which was correct pre-05.1-02
  but factually wrong once any plan 5.1-02..06 populates the registry.
  Tests 2 and 3 also missed a `runProductPipelines` stub in their
  orchestrator main() invocation, so the default iterator tried to
  run the real Power BI pipeline against a staged tmpdir that didn't
  have the spec.
- **Fix:** Test D rewritten to assert shape-invariant (`{name: string,
  run: function}` for every entry) plus a plan-specific pin (exactly
  one `powerbi` entry). Tests 2+3 gain a `runProductPipelines: async
  () => {}` no-op stub matching the pattern already used for runBetaPipeline
  et al. Tests of this plan (9, 10) also updated to import PRODUCT_PIPELINES
  directly from the leaf registry module (the orchestrator re-export
  triggers side-effect registration before any observable can check the
  "before" state).
- **Files modified:** `test/bin/generate-graph-client.test.mjs`,
  `test/bin/powerbi.test.mjs`
- **Commit:** `ceb3f5b`
