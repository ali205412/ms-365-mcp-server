# Phase 5.1 Plan 01: Shared runProductPipeline Contract + Orchestrator Integration Summary

**One-liner:** Ships a `runProductPipeline(opts)` shared contract plus the
`PRODUCT_PIPELINES` registry and step 4c/4d orchestrator slots that plans
5.1-02..06 will populate with Power Platform + M365 Admin product pipelines.

## What Was Built

**Task 1 — `bin/modules/run-product-pipeline.mjs` (new):**

A structurally-identical sibling of `runBetaPipeline` parameterised on
`{prefix, specUrl, specPath, snapshotPath, churnPolicy, churnEnvName,
openapiDir, generatedDir}`. Honors the same post-processing chain as
`beta.mjs` (hack.js import rewrite → `.strict()` → `.passthrough()` → errors
strip → HTML entity decode → function-path backtick fixup → prefix injection
→ MCP 64-char sha1-8 truncation). Merges via
`mergeBetaFragmentIntoClient` (reused unchanged — its regex-based schema
de-dup + endpoints-array splice already handles arbitrary `__<prefix>__`
namespaces).

Also exports `runProductChurnGuard(aliases, snapshotPath, policy, envName)`
which supports both D-04 policies:

- **permissive** (Power BI / Power Apps / Power Automate): additions silent,
  removals throw unless `envName === '1'`. Preview up to 10 removed names.
- **strict** (Exchange Admin REST v2 / SharePoint Tenant Admin): ANY delta
  (adds OR removes) throws unless `envName === '1'`. Preview up to 5
  removed + 5 added.

**Task 2 — orchestrator wiring:**

`bin/generate-graph-client.mjs` now exports `PRODUCT_PIPELINES = []` at the
module top with JSDoc documenting the sequential-merge invariant
(`mergeBetaFragmentIntoClient` reads + rewrites `client.ts`; two concurrent
merges corrupt the endpoints array). The `main()` deps bag accepts
`runProductPipelines` override; default iterates `PRODUCT_PIPELINES`
sequentially logging each entry name. Step 4c invokes the iterator after
the first `stubMissingSchemas` pass; step 4d re-runs stubs after product
merges to catch any `microsoft_graph_*` refs introduced by product
fragments (`stubMissingSchemas` is idempotent).

`.env.example` declares 5 new per-product churn env vars under a new
`region:phase5.1-codegen` block, each defaulting to `0` (fail-closed).

## Why It's Shaped This Way

**Why an empty registry + later-plan `.push(...)` over a registry built at
`main()` entry:** Plans 5.1-02..06 are independently plannable and
parallel-executable (05.1-CONTEXT D-01). If they had to modify
`generate-graph-client.mjs` directly, merge conflicts would serialise
them and defeat the parallelism. `PRODUCT_PIPELINES.push(...)` in a
per-product registration module (one file per product) keeps each plan's
diff to `bin/modules/<product>.mjs` + a one-line registration import.

**Why duplicate truncation + prefix-injection logic rather than extract
into a shared helper:** `beta.mjs` exports exactly two public symbols:
`runBetaPipeline` and `runChurnGuard`. Any extracted helper would need to
be re-exported and couple `run-product-pipeline.mjs` to beta-specific
internals. The helpers are ~10 lines each and functionally identical, so
duplication is the minimum-coupling option. If a third consumer ever lands
(unlikely — v1 codegen is the only other possible case and it lives in
`generate-mcp-tools.mjs`), a `post-process-helpers.mjs` factor-out is the
natural refactor.

**Why snapshot payload uses `ops` and not `beta_ops`:** The beta snapshot's
`beta_ops` field is product-specific; the per-product snapshot shape
should be generic so a single function signature handles all 5 products.
No cross-compatibility concern: each product has its own snapshot file
path (`bin/.last-<product>-snapshot.json`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing critical validation] Additional opts validation**

- **Found during:** Task 1 implementation
- **Issue:** Plan called out only `VALID_PREFIX_RE` early-check, but the
  opts bag had four other required fields (`churnPolicy`, `churnEnvName`,
  `specPath` + `snapshotPath`, `openapiDir` + `generatedDir`) that would
  cause obscure downstream failures if missing.
- **Fix:** Added type + non-empty-string guards for all five, each throwing
  with a message naming the missing/invalid field. Fail-fast at the top
  of the function rather than at `execSync`, `path.join`, or `fs.existsSync`.
- **Files modified:** `bin/modules/run-product-pipeline.mjs`
- **Commit:** `72744a5`

**2. [Rule 2 - Zero-aliases invariant] Guard against prefix-regex drift**

- **Found during:** Task 1 implementation
- **Issue:** If the prefix-injection regex ever stopped matching the
  upstream spec's operationIds (e.g. the spec started emitting uppercase-
  first aliases), `productAliases` would be an empty array and the churn
  guard would silently pass. This would leak unprefixed aliases into the
  main registry.
- **Fix:** Added a `productAliases.length === 0` check that throws with a
  message pointing at upstream spec drift or prefix-regex failure. The
  plan's security invariant ("codegen NEVER emits an unprefixed alias")
  is now positively enforced, not just asserted in tests.
- **Files modified:** `bin/modules/run-product-pipeline.mjs`
- **Commit:** `72744a5`

No plan-architectural deviations. No Rule 4 checkpoints required.

## Second stubMissingSchemas Pass Behavior

Against the fixture (3 ops, no `microsoft_graph_*` references), **the second
pass is a no-op**. Expected and correct: the fixture spec has no Graph type
overlap. Once plans 5.1-02..06 ship real Power Platform / EXO / SP-Admin
fragments, the second pass has a chance of stubbing Graph types shared
across workload boundaries (EXO mailbox schemas can cross-reference Graph
user types, for instance). The test asserts ordering and invocation count,
not stub count — matching the intent of the 05.1-RESEARCH anti-pattern
pin ("Do NOT skip stub pass after last merge").

## Auth Gates

None required — this is pure codegen pipeline work with no network calls
(fixture uses `specUrl: null` hand-authored path). No MSAL, no Graph API,
no Postgres, no Redis.

intentionally empty until plans 5.1-02..06 populate it — this is the
contract, not a stub.
