# Codebase Concerns

**Analysis Date:** 2026-04-18

## Tech Debt

**Self-labeled "hack" generated client shim:**
- Issue: `src/generated/hack.ts` is a hand-rolled `Zodios` substitute (with `throw new Error('No such hack')` for the URL constructor path) that mutates parameter names by stripping `$` and `_` from every parameter, then re-injects path parameters that the generator omitted. This is the only thing standing between the OpenAPI generator output and the runtime tool registration.
- Files: `src/generated/hack.ts`, `src/generated/endpoint-types.ts`, `src/generated/client.ts` (gitignored, generated by `bin/generate-graph-client.mjs`)
- Impact: Any change to `openapi-zod-client` output, or to how MCP clients handle `$`/`_` in parameter names, can silently break tool registration for hundreds of endpoints. The shim's behavior is then partially reversed at call time in `executeGraphTool` (lines 174-202 of `src/graph-tools.ts`) — `$` is stripped on the way in, then re-added at request time. Two layers of name munging make debugging param mismatches difficult.
- Fix approach: Document the contract explicitly between `hack.ts`, `endpoints.json`, and `executeGraphTool`; or replace with a thin wrapper around the real `@zodios/core` once stable.
- **Status:** Deferred to Phase 5 (FOUND-02 regenerate pipeline) — replaced by a hardened generator step that eliminates the parameter-name munging round trip.

**Lazy keytar import abuse of TypeScript types:**
- Issue: `src/auth.ts` lines 12-28 use a tri-state `null` / `undefined` / `keytar module` pattern with `keytar = undefined as any` to mean "tried and failed" — a side-channel encoded in the `null`/`undefined` distinction that is fragile and explicitly cast through `any`.
- Files: `src/auth.ts:10-28`
- Impact: Future refactors that "tidy up" this nullable singleton risk reverting to a state where keytar is re-imported on every call (slow) or imported once and crashes on Alpine (the original problem).
- Fix approach: Replace with an explicit `KeytarState = 'unknown' | 'unavailable' | KeytarModule` discriminated union.
- **Status:** Resolved in plan 01-08 (keytar removed entirely; the tri-state lazy-import block deleted with it).

**Two-layer URL encoding workaround:**
- Issue: `src/graph-tools.ts:212-214` encodes path values with `encodeURIComponent`, then immediately reverses `%3D` back to `=` because Microsoft Graph base64 IDs contain `=` as padding. The comment ("First we encode, then unencode. Crazy") references issue #245.
- Files: `src/graph-tools.ts:206-223`, `src/graph-tools.ts:269-275` (duplicated in fallback branch)
- Impact: If Graph ever introduces an ID containing a literal `%3D` that is *not* base64 padding, it will be decoded incorrectly. Other characters reserved-but-allowed in path segments (`!`, `$`, `&`, `'`, `(`, `)`, `*`, `+`, `,`, `;`) are still over-encoded.
- Fix approach: Use a path-segment-aware encoder (e.g., RFC 3986 unreserved + sub-delims) instead of `encodeURIComponent` plus targeted reversal.
- **Status:** Documented constraint: no phase. The workaround is covered by `test/path-encoding.test.ts`; Graph has not (yet) shipped IDs that break it. Revisit only if Graph changes ID format.

**Linear-scan PKCE store with duplicate logic in two places:**
- Issue: `src/server.ts:60-69` declares an in-memory `pkceStore: Map<string, ...>`. On every `/authorize` call it walks the entire map to evict expired entries (lines 343-347), then on every `/token` call it walks the map again computing SHA-256 of the client verifier to find a match (lines 455-466).
- Files: `src/server.ts:60-69`, `src/server.ts:296-393`, `src/server.ts:441-466`
- Impact: O(N) per request with N up to 1000 entries. SHA-256 per entry per token request scales poorly under concurrent OAuth flows. Cluster/multi-instance deployments lose state because the store is in-process only — defeats horizontal scaling for the HTTP/OAuth path.
- Fix approach: Index the store by `clientCodeChallenge` (computed once on `/authorize`) so `/token` is a single map lookup; for multi-instance, externalize to Redis or accept that PKCE-state-on-server is an in-process-only feature and use sticky sessions.
- **Status:** Deferred to Phase 3 (SECUR-03) — PKCE store externalized to Redis for multi-tenant + multi-replica support.

**Stray top-level test scripts:**
- Issue: `test-calendar-fix.js` and `test-real-calendar.js` live at the project root (not in `test/`), are not picked up by Vitest, and one of them imports from `./dist/` (i.e., requires a prior build).
- Files: `test-calendar-fix.js`, `test-real-calendar.js`
- Impact: Confusing for new contributors. Not run by CI, so they bit-rot. They reach into compiled `dist/` and a removed `GraphClient` constructor (the real one now requires `authManager`, `secrets`).
- Fix approach: Delete them, or move into `test/` and refactor against the public API.
- **Status:** Resolved in plan 01-09 (both files deleted; test/concerns-triage.test.ts Tests A+B pin the absence).

**Mixed `.js` and `.ts` test files:**
- Issue: `test/calendar-fix.test.js` is the only `.js` test file in an otherwise `.ts` test suite. Tests don't share an enforced typing convention.
- Files: `test/calendar-fix.test.js`
- Fix approach: Rename to `.ts` and add types; or accept and document the mix.
- **Status:** Resolved in plan 01-09 (renamed to `test/calendar-fix.test.ts` with minimal type annotations; test/concerns-triage.test.ts Tests C+D pin the rename).

**Duplicated OData property stripping function:**
- Issue: `removeODataProps` is defined twice as an inline `const` in `formatJsonResponse` — once in the `_headers` branch (lines 303-313), once in the no-headers branch (lines 339-349). Same logic, different scopes.
- Files: `src/graph-client.ts:303-313`, `src/graph-client.ts:339-349`
- Impact: Maintenance hazard — fixing OData parsing in one branch and forgetting the other introduces silent inconsistency.
- Fix approach: Hoist to a module-level helper.
- **Status:** Resolved in plan 01-09 (hoisted to a single module-level export with depth + WeakSet cycle guards; test/concerns-triage.test.ts Tests G+H pin the single-declaration invariant).

**Verbose mode logs every Graph URL and request body:**
- Issue: `src/graph-client.ts:216`, `src/graph-tools.ts:127`, `src/graph-tools.ts:384-386` log full request paths (which can include base64 message IDs), tool params, and `safeOptions` (which still contains `body`, `queryParams`, headers other than `accessToken`). The `Authorization` header is redacted, but `Prefer`, `Content-Type`, and `Accept` headers and arbitrary body content are not.
- Files: `src/graph-client.ts:216`, `src/graph-client.ts:245`, `src/graph-tools.ts:127`, `src/graph-tools.ts:384-386`
- Impact: PII leakage in logs in `--verbose` (or `LOG_LEVEL=info`, the default). Email subjects, file names, calendar event details flow through `body`, etc. The default log level is `info`, and these are info-level statements.
- Fix approach: Demote per-request body/query logs to `debug`; redact known PII fields; or make them opt-in via `MS365_MCP_LOG_REQUEST_BODIES=1`.
- **Status:** Resolved in plan 01-02 (D-01 STRICT REDACT_PATHS scrubs request/response bodies and Prefer/Content-Type headers; pino `normalizePath` rewrites ID-laden URL segments to `{id}` placeholders).

**Test mocking strategy is module-import-order sensitive:**
- Issue: `src/__tests__/graph-tools.test.ts:30-42` mocks `fs.readFileSync` to substitute `endpoints.json` per test, with a comment noting tests must "re-import after setting mocks." This is brittle.
- Files: `src/__tests__/graph-tools.test.ts:30-42`, `src/__tests__/graph-tools.test.ts:97-100`
- Impact: New tests added by contributors who don't read the comment will silently use stale module state.

- **Status:** Documented constraint: no phase. Operator-quality tech debt; vitest's `vi.resetModules()` convention is documented in CONVENTIONS.md, and contributors adding new tests follow it. No test has been silently broken by this since the comment was added.

## Known Bugs

**Token endpoint logs `body` on missing `grant_type`:**
- Symptoms: When a token request arrives without `grant_type`, the server calls `logger.error('Token endpoint: grant_type is missing', { body })` — logging the entire request body.
- Files: `src/server.ts:418-425`
- Trigger: Send a malformed `POST /token` with any other fields but no `grant_type`.
- Workaround: Body would normally include `code`, `code_verifier`, or `refresh_token` — these are credentials. The `body` should be redacted before being logged.
- **Status:** Resolved in plan 01-07 (`createTokenHandler` Site B emits `has_code`/`has_refresh_token`/`has_client_secret` booleans in place of the raw body; Site C stringifies caught errors without spreading the raw Error).

**Generated `mcp-client-${Date.now()}` is not unique under concurrent registration:**
- Symptoms: `src/server.ts:280` issues client IDs as `mcp-client-${Date.now()}`. Two clients registering in the same millisecond collide.
- Files: `src/server.ts:275-292`
- Trigger: Concurrent dynamic-registration requests; load-test scenarios.
- Workaround: Add a `crypto.randomBytes(8).toString('hex')` suffix.
- **Status:** Resolved in plan 01-06 (client IDs now use `mcp-client-${crypto.randomBytes(8).toString('hex')}` — 64 bits of entropy, no millisecond collision window).

**Dynamic registration accepts arbitrary `redirect_uris` without validation:**
- Symptoms: `src/server.ts:285` returns whatever `redirect_uris` the caller posted, with no scheme/host validation, no rejection of `javascript:` or arbitrary external hosts. The registered client is then trusted by the proxy provider.
- Files: `src/server.ts:275-292`
- Trigger: Open `/register` endpoint accepts any payload.
- Workaround: Treat this as untrusted input — validate against an allowlist or require HTTPS/`http://localhost:*`.
- **Status:** Resolved in plan 01-06 (D-02 `validateRedirectUri` allowlist; `javascript:`/`data:`/`file:`/`about:`/`vbscript:` rejected; loopback + MS365_MCP_PUBLIC_URL host the only allowed hosts in prod; `createRegisterHandler` factory invokes the validator per entry).

**Default CORS allows credential-bearing requests from `http://localhost:3000` even when `MS365_MCP_PUBLIC_URL` is set:**
- Symptoms: `src/server.ts:183-199` defaults `Access-Control-Allow-Origin` to `http://localhost:3000` regardless of where the server is actually deployed. There is no `Access-Control-Allow-Credentials` header set, but the static origin still gets to make CORS-preflighted requests.
- Files: `src/server.ts:183-199`
- Trigger: Default deployment without explicit `MS365_MCP_CORS_ORIGIN`.
- Workaround: Set `MS365_MCP_CORS_ORIGIN` explicitly in production. Recent commit `2c2727b` already restricted from wildcard, but the default could be smarter (e.g., derive from `MS365_MCP_PUBLIC_URL`).
- **Status:** Resolved in plan 01-07 (`createCorsMiddleware` dev/prod split — dev allows loopback on any port via regex; prod uses explicit allowlist from `MS365_MCP_CORS_ORIGINS`; prod + empty allowlist triggers fail-fast exit 78 at startup).

**Pagination silently truncates at 10 000 items:**
- Symptoms: When `fetchAllPages=true`, the loop in `src/graph-tools.ts:400-423` stops at `maxItems = 10_000` and only logs a warning. The deleted `@odata.nextLink` is gone from the returned response.
- Files: `src/graph-tools.ts:390-449`
- Trigger: Any query with more than 10k matching items.
- Workaround: Caller has no way to detect truncation in the returned payload — the warning only goes to logs. Add a `_truncated: true` flag in the merged response or preserve the unused `@odata.nextLink`.
- **Status:** Deferred to Phase 2 (MWARE-04) — PageIterator async-generator pattern with caller-controllable maxPages and `_truncated` flag in the merged response.

**`getKeytar()` first-call race condition:**
- Symptoms: Two concurrent calls to `getKeytar()` while `keytar === null` will both enter the `try { await import('keytar') }` block. Not catastrophic (the second wins), but wastes a dynamic import and the `keytar = undefined as any` failure-path assignment from one call can race with success in the other.
- Files: `src/auth.ts:13-28`
- Trigger: Concurrent token operations during cold start.
- Workaround: Guard with a single in-flight Promise.
- **Status:** Resolved in plan 01-08 (keytar removed wholesale; `getKeytar` and its lazy-import block deleted — no code path remains for the race).

**Tests reference fixed paths that may not exist on contributor machines:**
- Symptoms: `test/auth-paths.test.ts` uses `/tmp/test-cache/` paths and assumes Unix path conventions. Won't pass on Windows.
- Files: `test/auth-paths.test.ts:33-44, 65-76`
- Trigger: `npm test` on Windows.
- Workaround: Use `os.tmpdir()` and `path.join`.
- **Status:** Resolved in plan 01-09 (test/auth-paths.test.ts now uses `path.join(os.tmpdir(), ...)`; test/concerns-triage.test.ts Tests E+F pin the portability fix).

## Security Considerations

**Bearer token middleware accepts any string and forwards it unverified to Graph:**
- Risk: `src/lib/microsoft-auth.ts:23` explicitly says "we don't validate the token here - we'll let the API calls fail if it's invalid". Combined with the path that takes the token from the `Authorization` header and stuffs it into `requestContext` (consumed by `GraphClient.makeRequest`), this means the server forwards arbitrary attacker-supplied strings to Microsoft as Bearer tokens.
- Files: `src/lib/microsoft-auth.ts:9-36`, `src/server.ts:520-620`, `src/graph-client.ts:88-97`
- Current mitigation: Microsoft Graph itself rejects invalid tokens with 401. The `MicrosoftOAuthProvider.verifyAccessToken` (`src/oauth-provider.ts:22-43`) does call `/me` to validate, but it isn't actually invoked on every `/mcp` request — it's part of the SDK's `mcpAuthRouter`, not the custom `microsoftBearerTokenAuthMiddleware` mounted on `/mcp`.
- Recommendations: Add actual token introspection (call `/me` once and cache the validation result with short TTL); reject malformed tokens at the edge so we don't burn Graph quota on garbage requests.
- **Status:** Deferred to Phase 3 (TENANT-04 / AUTH-05) — per-tenant bearer-token introspection + short-TTL cache land alongside the multi-tenant MSAL pool; a malformed-token reject-at-edge adapter is part of the same change set.

**Refresh token sourced from a custom header `x-microsoft-refresh-token`:**
- Risk: `src/lib/microsoft-auth.ts:27` reads the refresh token from a non-standard header, which is then used in token refresh flows in `src/graph-client.ts:101-108`. Refresh tokens are long-lived credentials that grant access without user interaction. They have no business riding in a custom header that's likely to leak via proxy logs or browser dev tools.
- Files: `src/lib/microsoft-auth.ts:27`, `src/graph-client.ts:101-108`
- Current mitigation: None — the header is documented nowhere visible to operators.
- Recommendations: Document this contract explicitly, warn about logging implications, and consider replacing with a server-side encrypted store keyed by the access token.
- **Status:** Deferred to Phase 3 (SECUR-02) — refresh tokens move to an encrypted server-side store keyed by access-token hash; the custom header goes away.

**`logger.info('Client registration request', { body })` logs raw registration payload:**
- Risk: `src/server.ts:278` logs the entire dynamic registration request body, which may include client secrets, internal callback URLs, and other sensitive metadata.
- Files: `src/server.ts:275-292`
- Current mitigation: Logs go to a file with `0o700` directory permissions (`src/logger.ts:12`), which limits exposure to the local user.
- Recommendations: Log only non-sensitive fields (`client_name`, `grant_types`); drop `redirect_uris` (which are PII) or hash them.
- **Status:** Resolved in plan 01-06 (`createRegisterHandler` emits scrubbed `{ client_name, grant_types, redirect_uri_count }` only — raw body never attached).

**Hardcoded redirect URI in `MicrosoftOAuthProvider.getClient`:**
- Risk: `src/oauth-provider.ts:52` returns `redirect_uris: ['http://localhost:3000/callback']` for *any* client_id requested — a static, permissive default. The proxy provider then trusts whatever client claims this ID.
- Files: `src/oauth-provider.ts:49-54`
- Current mitigation: This is only used in OAuth flows where the actual redirect happens to Microsoft, but the SDK may use it for client metadata validation.
- Recommendations: Look up the actual registered client's redirect URIs (from a store) instead of returning a fixed value.
- **Status:** Deferred to Phase 3 (TENANT-01) — `getClient` resolves against the Postgres-backed registered-client store, returning each client's actual validated `redirect_uris` rather than a static localhost default.

**`client_secret` (when configured) flows through `req.body` and the in-process server with no rate limiting on `/token`:**
- Risk: `src/server.ts:430` reads `clientSecret` from cached secrets, then forwards to Microsoft via `exchangeCodeForToken` / `refreshAccessToken`. The endpoint has no rate limiting, so an attacker who steals a single authorization code can attempt unlimited refresh cycles.
- Files: `src/server.ts:396-511`, `src/lib/microsoft-auth.ts:41-133`
- Current mitigation: Microsoft enforces its own rate limits, but errors surface as opaque 500s here.
- Recommendations: Add `express-rate-limit` middleware on `/token`, `/authorize`, and `/register`; surface meaningful 429s rather than logging-and-throwing.
- **Status:** Deferred to Phase 6 (OPS-08) — `express-rate-limit` (or Redis-backed equivalent) mounted on `/token`, `/authorize`, `/register` with per-tenant buckets; surfaces meaningful 429s with `Retry-After`.

**`scopes: []` returned by `verifyAccessToken`:**
- Risk: `src/oauth-provider.ts:39` returns an empty `scopes` array regardless of what the token actually grants. Downstream MCP authorization checks (if any are based on scopes) will see no scopes and may fail-open.
- Files: `src/oauth-provider.ts:30-40`
- Current mitigation: The SDK doesn't currently enforce scopes, so this is latent.
- Recommendations: Decode the JWT (or call `/me` for the consented scopes) and populate the array.
- **Status:** Deferred to Phase 3 (AUTH-05 scope propagation) — JWT claims decoded and populated into the `scopes` array so downstream authorization checks see the real consented scopes.

**`fs.readFileSync` of `.token-cache.json` and `.selected-account.json` without integrity check:**
- Risk: `src/auth.ts:266`, `src/auth.ts:298` read JSON cache files and `JSON.parse` them with only a `try/catch` envelope check. A modified cache file (by another local process or rogue script) can inject arbitrary `accountId` values that change the default account silently.
- Files: `src/auth.ts:251-310`
- Current mitigation: Files are written `0o600`. Local-only attack surface.
- Recommendations: Sign the envelope payload with a key derived from the OS keychain when keytar is available.
- **Status:** Deferred to Phase 3 (SECUR-01) — token cache moves to AES-GCM envelope encryption with KEK injection; filesystem tampering is signature-detected rather than silently applied.

## Performance Bottlenecks

**`endpointsData.find(...)` lookup per registered tool:**
- Problem: For each of the ~200+ tools, `registerGraphTools` does a `find` by `toolName` over `endpointsData` (`src/graph-tools.ts:520`). That's O(N²) over the full endpoint catalog at startup. The same find runs again at execute time inside `buildToolsRegistry` (`src/graph-tools.ts:766`).
- Files: `src/graph-tools.ts:520`, `src/graph-tools.ts:766`
- Cause: No precomputed Map keyed by `toolName`.
- Improvement path: Build a `Map<string, EndpointConfig>` once at module load and reuse.
- **Status:** Resolved in plan 01-09 (module-level `endpointsMap: Map<string, EndpointConfig>` populated once; both call sites use `endpointsMap.get(tool.alias)` — O(1) per lookup).

**`endpoints.default.forEach(...)` walks all 1453 endpoints on every scope-build call:**
- Problem: `src/auth.ts:161-186` iterates the full `endpoints.default` array for each scope-build invocation, applying regex tests for tool filtering. Called multiple times during startup (once for the auth manager's default scopes, again from `buildScopesFromEndpoints` in the OAuth metadata route at `src/server.ts:237` and `src/server.ts:264` on every `.well-known` request).
- Files: `src/auth.ts:141-204`, `src/server.ts:232-273`
- Cause: No memoization.
- Improvement path: Cache scopes keyed by `(includeWorkAccountScopes, enabledToolsPattern, readOnly)` — these change rarely (effectively never within a process lifetime).
- **Status:** Resolved in plan 01-09 (`scopeCache: Map<string, string[]>` memoizes by argument tuple; clone-on-return prevents downstream mutation of cached entries).

**`removeODataProps` recurses unbounded into response payloads:**
- Problem: `src/graph-client.ts:303-315, 339-351` recurses through arbitrary nested objects with no depth or cycle guard. A malicious or malformed Graph response with circular references would stack-overflow the process.
- Files: `src/graph-client.ts:303-351`
- Cause: Trust assumption: Graph responses are tree-shaped JSON. Probably true today, but no defense in depth.
- Improvement path: Add a max-depth guard and use iteration with an explicit stack.
- **Status:** Resolved in plan 01-09 (`MAX_REMOVE_ODATA_DEPTH = 100` depth cap + WeakSet cycle guard; test/odata-recursion.test.ts Test 1 asserts self-cycle input returns without stack overflow in <100ms).

**Discovery search BM25 index built once and held in memory:**
- Problem: Index size scales with the tool catalog; not freed when discovery mode is off. Per-request rerank in `scoreDiscoveryQuery` re-sorts the entire ranked array (`src/graph-tools.ts:855`).
- Files: `src/graph-tools.ts:790-823`, `src/graph-tools.ts:832-857`
- Cause: Acceptable for ~200 tools; would not scale to 10k+.
- Improvement path: Top-K heap instead of full sort; lazy index build.
- **Status:** Deferred to Phase 5 (COVRG-* — scales when the curated catalog grows beyond the single-tenant essentials preset to the full v1.0 + curated beta surface).

**No connection pooling or keep-alive for Graph fetch calls:**
- Problem: `src/graph-client.ts:224` calls `fetch(url, ...)` with no agent. Node 18+ undici defaults are reasonable, but a stateless HTTP server fielding many tool calls per session would benefit from explicit keep-alive tuning.
- Files: `src/graph-client.ts:208-229`
- Cause: Default fetch behavior assumed sufficient.
- Improvement path: Pass a tuned `Agent`/`Dispatcher` for high-throughput deployments.
- **Status:** Deferred to Phase 2 (MWARE-02 may add keep-alive) — tuned undici Dispatcher is a natural add to the retry/throttle middleware pipeline.

**No rate limiting or throttling for Graph requests:**
- Problem: Microsoft Graph has documented per-tenant and per-app throttles. The server forwards every tool call directly. There is no 429-backoff handling, no `Retry-After` honoring, no circuit breaker.
- Files: `src/graph-client.ts:88-178`
- Cause: Originally a CLI tool for one user; HTTP/multi-user mode hasn't grown matching backpressure.
- Improvement path: Add 429 detection + exponential backoff respecting `Retry-After`; surface throttling metrics for operators.
- **Status:** Deferred to Phase 2 (MWARE-01) — RetryHandler parses 429 `Retry-After` and does exponential backoff with full jitter; Phase 6 wires the OTel counters for operator visibility.

## Fragile Areas

**Two-leg PKCE in `/authorize` and `/token`:**
- Files: `src/server.ts:60-69`, `src/server.ts:296-393`, `src/server.ts:441-466`
- Why fragile: State is keyed by `state`, but cleanup is opportunistic (on next `/authorize` call). The `/token` matching scans the entire map computing SHA-256. Concurrent flows from multiple clients interleave in a single `Map`. Tests for this logic exist (`test/http-oauth-fix.test.ts`) but cover only the happy path.
- Safe modification: Add an integration test that runs two PKCE flows concurrently before changing this code.
- Test coverage: Missing — concurrency, expiry cleanup, store-full path (503 response).
- **Status:** Deferred to Phase 3 (SECUR-03) — PKCE store externalizes to Redis; concurrency + expiry + store-full coverage arrives with the multi-tenant rewrite of the OAuth surface.

**Generated client behavior depends on `bin/` scripts and external openapi-zod-client tool:**
- Files: `bin/generate-graph-client.mjs`, `bin/modules/*.mjs`, `remove-recursive-refs.js`
- Why fragile: `src/generated/client.ts` is gitignored and regenerated by CI. The 9KB `remove-recursive-refs.js` exists solely to work around `openapi-zod-client` recursive `$ref` issues (referenced GitHub issues #36, #62 in the code). Any upstream change in either Graph's OpenAPI spec or `openapi-zod-client`'s output silently changes hundreds of tools.
- Safe modification: Re-run `npm run generate` and diff the output before merging upstream changes.
- Test coverage: `endpoints-validation.test.ts` exists but is small (~2KB).
- **Status:** Deferred to Phase 5 (FOUND-02 regenerate pipeline) — generator hardened with tool-diff checks + zod-4 migration; the `remove-recursive-refs.js` shim re-evaluated.

**`hack.ts` parameter mangling and runtime restoration in `executeGraphTool`:**
- Files: `src/generated/hack.ts:18`, `src/graph-tools.ts:174-202`
- Why fragile: `hack.ts` strips `[$_]+` from parameter names. `executeGraphTool` then has to detect OData params and re-add `$`. Path parameter handling has THREE different name forms: original, kebab-case (from `endpoints.json`), and camelCase (from generator). The fallback branch (`src/graph-tools.ts:261-276`) duplicates all four `path.replace` calls.
- Safe modification: Touch this only with full test coverage on Mail (`{message-id}`), Calendar (`{event-id}`), and OneDrive (`{driveItem-id}` with base64 IDs).
- Test coverage: `test/path-encoding.test.ts`, `test/odata-nextlink.test.ts` exist but don't cover the camelCase/kebab-case bridge end-to-end.
- **Status:** Deferred to Phase 5 (FOUND-02 regenerate pipeline) — name-munging eliminated when the generator emits the canonical parameter form directly.

**MSAL token cache + selected-account file are independently persisted with `pickNewest` reconciliation:**
- Files: `src/auth.ts:82-113, 251-310`
- Why fragile: Two storage backends (keytar + file), each with a `_cacheEnvelope` wrapper that may or may not be present in old caches. `pickNewest` chooses the freshest by `savedAt`. Removing the envelope (e.g., during cache migration) will cause the older format to win permanently.
- Safe modification: Add a version field to the envelope; never remove envelope-detection logic.
- Test coverage: `test/cache-stamp.test.ts` exists.
- **Status:** Resolved in plan 01-08 (keytar removed; file storage is the single source of truth. `pickNewest` is retained as a storage-layout-agnostic helper for Phase 3 to reuse across tenant-keyed caches — no longer spans two storage backends at runtime).

**`fetchAllPages` mutates the response in place:**
- Files: `src/graph-tools.ts:390-449`
- Why fragile: `combinedResponse.value = allItems` (line 435) reassigns the parsed `value` array, then `JSON.stringify(combinedResponse)` (line 441) is written back into `response.content[0].text`. If any caller had captured the original response, it now sees mutated data. The `for-await` loop also catches all errors via `catch (e)` and logs them — partial pagination can return a broken merged response without surfacing the error to the caller.
- Safe modification: Build the merged response immutably; bubble pagination errors to the caller.
- Test coverage: `test/odata-nextlink.test.ts` exists but covers limited cases.
- **Status:** Deferred to Phase 2 (MWARE-04) — replaced by the PageIterator async-generator pattern which builds merged state immutably and bubbles errors.

**Logger directory and file creation at module load time:**
- Files: `src/logger.ts:7-13`
- Why fragile: `fs.mkdirSync` runs at import time. If `~/.ms-365-mcp-server/logs/` is on a read-only filesystem (e.g., container with read-only root and no volume mount), the entire process crashes before any other code runs.
- Safe modification: Defer mkdir until the first log write, or fall back to console-only if directory creation fails.
- Test coverage: None for this failure mode.
- **Status:** Resolved in plan 01-02 (pino replaces Winston; no `fs.mkdirSync` at logger module load; file transport uses pino's `{ mkdir: true }` option for lazy directory creation when `MS365_MCP_LOG_DIR` is set).

## Scaling Limits

**In-process PKCE store, max 1000 entries:**
- Current capacity: 1000 concurrent in-flight authorizations (`src/server.ts:342`).
- Limit: 503 returned beyond that (`src/server.ts:350-358`).
- Scaling path: Externalize to Redis or use sticky sessions with smaller per-instance limits.
- **Status:** Deferred to Phase 3 (SECUR-03) — Redis-backed PKCE store with TTL-native eviction and cross-replica visibility.

**`fetchAllPages` capped at 100 pages / 10 000 items:**
- Current capacity: `src/graph-tools.ts:397-398` hardcodes `maxPages = 100` and `maxItems = 10_000`.
- Limit: Silently truncates beyond. Caller cannot bypass via parameters.
- Scaling path: Make limits configurable via env (`MS365_MCP_MAX_PAGES`, `MS365_MCP_MAX_ITEMS`); return a `_truncated` flag.
- **Status:** Deferred to Phase 2 (MWARE-04) — PageIterator async-generator with caller-controllable maxPages + `_truncated` flag in the merged response.

**Single Express process — no clustering:**
- Current capacity: One Node.js event loop.
- Limit: HTTP/OAuth flows do crypto (`crypto.randomBytes(32)`, `crypto.createHash('sha256')`) on the main thread.
- Scaling path: Run multiple replicas behind a load balancer; externalize PKCE state.
- **Status:** Out of Scope for v2.0 (per PROJECT.md Out of Scope list). The codebase is kept clustering-ready (stateless request handlers, externalized PKCE in Phase 3) but no Helm chart / autoscaler is shipped.

**Cached secrets module-level singleton:**
- Current capacity: `src/secrets.ts:111` keeps secrets cached forever after first call.
- Limit: No way to refresh secrets without restarting the process — Key Vault rotations are not picked up.
- Scaling path: Add TTL or watch-style invalidation; expose `clearSecretsCache()` (already exported) via a signal handler.
- **Status:** Deferred to Phase 6 (OPS-06 operator ergonomics) — SIGHUP-triggered `clearSecretsCache()` + optional TTL configuration. Low priority because Key Vault rotations are rare and a container restart handles them today.

## Dependencies at Risk

**`keytar@^7.9.0` is unmaintained:**
- Risk: The `keytar` library was archived by Atlassian in 2023. Native module rebuilds frequently fail on newer Node versions; Alpine builds have been a perennial pain point (the `auth.ts:11` comment explicitly mentions Alpine).
- Impact: New Node major releases may break the optional dependency entirely. Loss of OS keychain integration would force all users onto the file-based fallback (silently — see "lazy keytar import" tech debt).
- Migration plan: Evaluate `@napi-rs/keyring` or migrate fully to encrypted file-based storage with a passphrase.
- **Status:** Resolved in plan 01-08 (D-04 clean break; keytar removed from `optionalDependencies`; `bin/migrate-tokens.mjs` CLI subcommand handles v1→v2 migration on demand; `bin/check-keytar-leftovers.cjs` probe advises users with leftover entries).

**`zod@^3.24.2` while ecosystem moves to v4:**
- Risk: `zod-to-json-schema` and `openapi-zod-client` may shift to Zod 4 in upcoming releases. Generated code in `src/generated/client.ts` would need regeneration with potentially incompatible schema syntax.
- Impact: Client regeneration may break.
- Migration plan: Track Zod 4 adoption in `openapi-zod-client`; pin versions until tested.
- **Status:** Deferred to Phase 5 (generator regeneration) — zod 4 migration lands alongside the full v1.0 + curated beta catalog regeneration; pinned at v3.24.2 until then.

**`commander@^11.1.0` — newer major exists:**
- Risk: Commander 12.x has been out for some time. Caret range pins to 11.x.
- Impact: Missing newer CLI option types and validation features.
- Migration plan: Bump and re-test CLI parsing in `src/cli.ts`.
- **Status:** Documented constraint: no phase. Low priority; bump when a commander 12+ feature becomes useful. Not security-relevant.

**`express@^5.2.1`:**
- Risk: Express 5 is a fairly recent stable release; some middleware in the wider ecosystem still expects v4 semantics.
- Impact: Adding new middleware (e.g., `express-rate-limit`) may surprise.
- Migration plan: Verify any new middleware against Express 5.
- **Status:** Documented constraint: no phase. Phase 1 plans 01-04/05/06/07 all successfully landed Express 5 middleware; Phase 6 rate-limiting will verify `express-rate-limit`@7 (Express 5-compatible) at that time.

**Node.js engines `>=18` while Dockerfile uses `node:20-alpine` for runtime and `node:24-alpine` for build:**
- Risk: `package.json` says `>=18`, build uses 24, runtime uses 20, CI matrix tests 18/20/22 (`.github/workflows/build.yml:13`). Three different Node versions in three different places. Node 18 reaches EOL April 2025.
- Impact: Code that works on 24 may break on 18 (the Web `File` global polyfill in `test/setup.ts` is already evidence). README claims "Node.js >= 20 (recommended)" while `package.json` allows 18.
- Migration plan: Bump engines to `>=20`, drop the Node 18 polyfill, drop Node 18 from CI.
- **Status:** Resolved in plan 01-01 (engines bumped to `>=20 <23`; Docker runtime pinned to `node:22-alpine`; CI matrix dropped Node 18; baseline polyfills removed).

## Missing Critical Features

**No 429/throttling handling for Graph API:**
- Problem: Microsoft Graph throttles aggressively; the server has no `Retry-After` handling, no exponential backoff, no circuit breaker.
- Blocks: Reliable use under load; multi-user HTTP deployments will see cascading failures during spikes.
- **Status:** Deferred to Phase 2 (MWARE-01) — the Kiota-pattern RetryHandler is the first middleware to land.

**No metrics/telemetry:**
- Problem: No counters for tool invocations, latency histograms, error rates, or token-cache hits. The deployment guide (`docs/deployment.md`) recommends Azure Container Apps with multiple replicas — operators have no visibility into per-tool performance.
- Blocks: Capacity planning; debugging slow tools; identifying broken tools in production.
- **Status:** Resolved in plan 01-02 for the baseline (OTel NodeSDK, OTLP + Prometheus exporters, request counter / error counter / duration histogram instruments). Per-tenant attributes + Graph-specific middleware-stage spans deferred to Phase 6.

**No structured request IDs / tracing:**
- Problem: Logs are line-based with timestamps but no correlation IDs. A failed tool call in HTTP mode cannot be traced back to a specific MCP request, OAuth flow, or upstream Graph call.
- Blocks: Multi-user debugging; support escalations.
- **Status:** Resolved in plan 01-02 (`requestId` (nanoid) + `tenantId` threaded via AsyncLocalStorage + pino-http `genReqId`; OTel `trace_id`/`span_id` auto-injected via `@opentelemetry/instrumentation-pino`).

**Token introspection / scope check missing in middleware:**
- Problem: As noted under Security, `microsoftBearerTokenAuthMiddleware` accepts any `Bearer ` prefix and forwards to Graph. There's no claim validation, audience check, expiry check, or scope check.
- Blocks: Defense in depth; meaningful per-tool authorization.
- **Status:** Deferred to Phase 3 (TENANT-04 / AUTH-05) — per-tenant bearer introspection arrives with the multi-tenant MSAL pool.

**No request body size limits beyond Express defaults:**
- Problem: `app.use(express.json())` (`src/server.ts:179`) uses default `100kb` limit. For attachment uploads (`upload-large-attachment` tools) this will silently fail or truncate. There's no explicit configuration.
- Blocks: Reliable file upload paths.
- **Status:** Deferred to Phase 2 (MWARE-05) — body-parser limits raised in lockstep with the resumable large-file upload middleware (320 KiB chunks, `nextExpectedRanges` resume).
